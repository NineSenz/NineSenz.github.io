    React事件和原生事件的区别

    九练 7/5/2021 react

    # 什么是事件

    首先说JS就是为了实现一些动态的操作，而用户会有时候想去实现一些功能，如提交表单、鼠标点击等，就要在浏览器中触发这个事件，然后浏览器会感知(或者说捕获)到用户的这个行为就会去响应处理这个事件，这个就称之为事件

    # 什么是事件对象

    系统在调用处理程序时，把事件会发生的一切信息，都封装成一个对象，然后作为一个参数传递给事件处理程序，而这个对象就是事件对象。

    在原生函数中，经常会看见一个event的东西，而这个东西就是我们说的事件对象

    # 原生事件和react中的事件区别

    react中的事件时绑定到document上面的，而原生事件时绑定到dom上面的，因此相对绑定的地方来说，dom上的事件要优先于document上的事件执行

    # react事件处理上的优点

    • 几乎所有的事件都有代理(delegate)到document,达到性能优化的目的
    • 对于每种类型的事件，统一使用分发函数(dispatchEven)分发事件
    • 事件对象(event)是合成对象(syntheticEvent), 不是原生的

    # React合成事件

    # 为什么会抽象为合成事件？

    如果把所有的事件处理函数都绑定在DOM上，那么在页面响应的时候就会受到影响，导致页面很慢，react为了避免这类DOM事件的滥用，同时屏蔽底层之间不同浏览器事件的系统差异，实现了一个中间层 -- syntheticEvent

    # 原理

    在react中，如果需要绑定事件，一般都会在JSX中这么写：

      <div onClick={this.onClick}>我是react点击事件</div>
    1

    但是在react中，并不是把该Click事件真实的绑定在div的DOM上，而是绑定在了document上，当事件发生并且通过冒泡的方式冒泡至document时，react才会将事件的内容交给相对应的函数去处理

    # 在React中使用原生事件

    虽然react几乎封装了所有的原生事件，但有以下场景不得不使用原生事件来进行业务逻辑处理：

    • Modal开启以后，点击其他空白区域需要关闭Modal
    • 引入一些以原生事件实现的第三方库，并且相互之间需要有交互

    由于原生事件需要绑定在真实的DOM上，所以一般在compoentDidMount \ ref函数执行阶段进行绑定。

    # 原生事件和合成事件混合使用

    如果在业务场景中，需要混合使用原生事件和合成事件，那么在使用过程中，需要注意如下几点：

    # 响应的顺序

  onDOMClick = (e) => {

  }

  onReactClick = (e) => {

    原因分析： 首先我们知道原生事件时绑定在DOM上面的，而合成事件时绑定在document上面的，因此DOM上面的事件先被冒泡，则先执行，然后再冒泡到document, 合成事件才被执行

    # 阻止冒泡

    如果在onDOMClick事件中调用了e.stopPropagation()

    onDOMClick = (e) => {
        console.log('dom event!')
        e.stopPropagation()
    }

    这时候由于阻止了冒泡，则不会冒泡到document，则合成事件就不会被执行，因此只会输出 dom event!

    在举一个复杂一点的例子，如果这个例子都懂了，说明你对react中的合成事件和原生事件有了更进一步的学习

    用原生的解释可以说是调用了阻止冒泡的函数之后，在该函数后面要执行的函数就不会被冒泡到，也就不会执行，但是在react中，是在调用了stopPropagtion之后，就会给自己添加一个标识isPropagtionStopped来确定后面的监听器是否执行

    结论：

    合成事件统一注册在document上面，所以原生事件总是比合成事件先执行 阻止原生事件的冒泡后，会阻止合成事件的执行
